# Trazador — Linear-Native Agent Workflow

Trazador connects Claude/Codex agents to Linear so that every task is traceable from requirement to shipped code.

## Core Principle

**Linear is the single source of truth.** No local docs/plans/ or docs/brainstorms/ files.
All specs, plans, progress, and reviews live as Linear issues and comments.

## Prerequisites

Trazador bundles the **Linear MCP server** (`linear-server`) via `.mcp.json`.
On first use, Linear's OAuth flow will prompt you to authorize via browser.
If the MCP is not loading, `/trazador:init` will help you configure it.

## Configuration

Project-specific settings live in `.trazador/config.yaml`, generated by `/trazador:init`.

All commands read workspace constants (team, project, statuses, labels) from this config file.
**You must run `/trazador:init` once per project before using other commands.**

## Commands

| Command | Purpose | When to use |
|---------|---------|-------------|
| `/trazador:init` | One-time setup — configure Linear workspace, validate MCP | First time using trazador in a project |
| `/trazador:discovery` | Project reality check — analyze code vs Linear, find gaps, define scope | Periodic assessment or when unsure what to build |
| `/trazador:plan` | PM intake — define a feature, break it down, create Linear issues | You have a new feature idea or goal |
| `/trazador:work` | Agent execution — pick up a Linear issue and implement it | A `Todo` issue is ready for implementation |
| `/trazador:review` | Goal-aligned review — validate work against the original task | PR is ready, before merge |
| `/trazador:sync` | Automatic state sync — update Linear from agent actions | Runs automatically after plan/work/review |
| `/trazador:autopilot` | Autonomous supervisor — dispatch teammate agents to work through issues | Hands-free execution of Todo backlog |

## Workflow

```
You (PM)                    Agent (Claude/Codex)
   |                              |
   |-- /trazador:init ----------->|  (one-time setup)
   |                              |-- validates Linear MCP
   |                              |-- creates .trazador/config.yaml
   |                              |
   |-- /trazador:discovery ------>|  (periodic)
   |                              |-- scans code + Linear
   |                              |-- finds gaps, proposes scope
   |<- dialogue ------------------|
   |                              |-- creates backlog issues
   |                              |
   |-- /trazador:plan "feature" ->|
   |                              |-- asks clarifying questions
   |<- answers -------------------|
   |                              |-- creates Linear issue(s)
   |                              |-- sync: status -> Todo
   |                              |
   |-- /trazador:work ISSUE-XX -->|
   |                              |-- reads issue + comments
   |                              |-- sync: status -> In Progress
   |                              |-- implements, tests, commits
   |                              |-- creates PR
   |                              |-- sync: status -> In Review, links PR
   |                              |
   |-- /trazador:review ISSUE-XX >|
   |                              |-- reads issue spec + PR diff
   |                              |-- validates goal alignment
   |                              |-- runs technical review agents
   |                              |-- posts findings as Linear comment
   |                              |-- sync: status -> Done (if approved)
   |                              |
   |-- /trazador:autopilot ------>|  (hands-free)
   |                              |-- supervisor picks Todo issues
   |                              |-- spawns teammate agent per issue
   |                              |-- teammate: work + self-review
   |                              |-- supervisor logs results
   |                              |-- repeats until done or max reached
```

## Agent Session Protocol

Every agent session MUST:

1. Read `.trazador/config.yaml` for workspace constants
2. Check Linear for assigned/in-progress issues
3. If resuming work, read the issue and its comments for context
4. Never create local execution trackers — use Linear
5. On completion, always run sync to update Linear state

## Reading Config

All commands must read `.trazador/config.yaml` at the start to get:
- `linear.team_name` and `linear.team_key` — the Linear team
- `linear.project_name` — the Linear project to use
- `linear.statuses.*` — mapped status names
- `linear.labels` — available labels
- `workflow.source_of_truth` — "linear-only" or "hybrid"

**Never hardcode team/project names.** Always read from config.

## Sync Events

Trazador sync fires automatically on these transitions:

| Event | Linear Update |
|-------|--------------|
| Plan created | New issue in Todo with spec |
| Work started | Status -> In Progress, comment with branch name |
| Commit pushed | Comment with commit summary |
| PR created | Status -> In Review, link PR to issue |
| Review passed | Comment with review summary |
| Review failed | Comment with findings, status stays In Review |
| PR merged | Status -> Done, closing comment |

## Design Principles

1. **Stateless agents** — Any session can pick up any issue. Context comes from Linear.
2. **PM stays in control** — Agents never auto-assign or auto-start without explicit trigger.
3. **Simplest path** — Find the minimal implementation that meets acceptance criteria.
4. **Goal alignment** — Every review validates against the original issue spec, not just code quality.
5. **No local artifacts** — Plans, specs, and progress live in Linear. Code lives in git.
