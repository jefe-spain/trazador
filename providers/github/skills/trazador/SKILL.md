# Trazador — GitHub Issues Agent Workflow

Trazador connects Claude/Codex agents to GitHub Issues so that every task is traceable from requirement to shipped code.

## Core Principle

**GitHub Issues is the single source of truth.** No local docs/plans/ files.
All specs, plans, progress, and reviews live as GitHub issues and comments.

## Prerequisites

Trazador bundles the **GitHub MCP server** (`github`) via `.mcp.json`.
You need a `GITHUB_TOKEN` environment variable with repo access.

## Configuration

Project-specific settings live in `.trazador/config.yaml`, generated by `/trazador:init`.

All commands read workspace constants from this config file.
**You must run `/trazador:init` once per project before using other commands.**

## Commands

| Command | Purpose | When to use |
|---------|---------|-------------|
| `/trazador:init` | One-time setup — configure GitHub workspace | First time |
| `/trazador:discovery` | Project reality check — code vs issues | Periodic |
| `/trazador:plan` | Define a feature, create GitHub issues | New feature |
| `/trazador:work` | Pick up an issue and implement it | Issue is ready |
| `/trazador:review` | Validate implementation against spec | PR is ready |
| `/trazador:sync` | State sync — update labels and comments | Automatic or manual |
| `/trazador:autopilot` | Autonomous supervisor — dispatch agents | Hands-free |

## Status Tracking

GitHub Issues doesn't have built-in workflow states like Linear or Jira. Trazador uses **labels** to track status:

| Status | Label | Notes |
|--------|-------|-------|
| Backlog | `backlog` | Open issue, low priority |
| Todo | `todo` | Open issue, ready for work |
| In Progress | `in-progress` | Agent is working on it |
| In Review | `in-review` | PR created, awaiting review |
| Done | (closed) | Issue closed when PR merged |

## Agent Session Protocol

Every agent session MUST:

1. Read `.trazador/config.yaml` for workspace constants
2. Check GitHub for assigned/in-progress issues
3. Never create local execution trackers — use GitHub
4. On completion, update labels and post comments

## Design Principles

1. **Stateless agents** — Context comes from GitHub Issues.
2. **PM stays in control** — Explicit triggers only.
3. **Simplest path** — Minimal code to meet acceptance criteria.
4. **Goal alignment** — Reviews validate against the original issue spec.
5. **No local artifacts** — Plans in GitHub, code in git.
