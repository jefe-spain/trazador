# Trazador Init — Project Setup Wizard (Jira)

One-time interactive setup that binds this repository to a Jira workspace and configures the trazador workflow.

## Input

<init_input> #$ARGUMENTS </init_input>

Arguments are ignored — this command is always interactive.

## Pre-flight Check

Before starting, check if `.trazador/config.yaml` already exists:

1. Read `.trazador/config.yaml`
2. If it exists and has valid content:
   - Show the current configuration to the user
   - Ask: "Trazador is already configured. Do you want to **reconfigure** or **exit**?"
   - If exit, stop here
   - If reconfigure, proceed with the wizard (existing values become defaults)

## Execution Flow

### Phase 1: Ensure Jira MCP is Available

Trazador requires the Jira MCP server (`jira-server`). This phase ensures it's configured and working.

**Step 1: Check if the MCP is loaded**

The trazador plugin bundles Jira MCP via its `.mcp.json`. It should be available automatically.

1. Attempt to call `mcp__jira-server__list_projects()`
2. **If it succeeds:**
   - Store the available projects
   - Tell the user: "Jira MCP is connected. Found N project(s)."
3. **If it fails:**
   - The bundled MCP may not have loaded, or OAuth hasn't been completed.
   - Tell the user:
     ```
     Jira MCP is not responding.

     The trazador plugin bundles the Jira MCP server, but it may need authorization.
     On first use, you'll be prompted to authorize via browser.

     If the plugin MCP didn't load, you can add it manually:
       claude mcp add --transport http jira-server https://mcp.atlassian.com/v1/mcp

     Or add to your project's .mcp.json:
       { "mcpServers": { "jira-server": { "type": "http", "url": "https://mcp.atlassian.com/v1/mcp" } } }
     ```
   - Ask: "Would you like to retry, continue without Jira (config only), or stop?"
   - If retry, go back to the test
   - If continue, mark `jira_mcp_verified: false` and skip to Phase 5
   - If stop, exit gracefully

### Phase 2: Select Jira Project

**If MCP is available:**

1. List available projects from the MCP
2. Present the list of projects to the user
3. Ask: "Which project will this workspace track?" (single choice)

**Store:** `project_name`, `project_key`, `project_id`

### Phase 3: Verify Statuses & Labels

1. Fetch project workflow statuses from the MCP
2. Map trazador standard statuses to Jira equivalents:
   - `backlog` → "Backlog" or "Open"
   - `todo` → "To Do" or "Selected for Development"
   - `in_progress` → "In Progress"
   - `in_review` → "In Review" or "Code Review"
   - `done` → "Done" or "Closed"
   - If any are missing, warn the user and list what's available
   - Let the user map the closest equivalents
3. Fetch issue types and labels/components

**Store:** `statuses` (mapped names), `issue_types`, `labels`

### Phase 4: Configure Source of Truth

Ask the user with **AskUserQuestion**:

> How should this project manage specs and plans?
>
> 1. **Jira-only** (recommended) — All specs, plans, and progress live in Jira. No local docs/plans/ files.
> 2. **Hybrid** — Jira for tracking, but allow local docs/ for architecture notes and research.

**Store:** `source_of_truth: "jira-only" | "hybrid"`

### Phase 5: Write Configuration

Create `.trazador/config.yaml` with:

```yaml
# Trazador Configuration
# Generated by /trazador:init on YYYY-MM-DD
# PM Tool: Jira

tool: jira
mcp_server: jira-server

jira:
  mcp_verified: true|false
  project_name: "<project>"
  project_key: "<KEY>"
  project_id: "<id>"
  statuses:
    backlog: "Backlog"
    todo: "To Do"
    in_progress: "In Progress"
    in_review: "In Review"
    done: "Done"
  issue_types:
    - "Story"
    - "Bug"
    - "Task"
  labels:
    - "Feature"
    - "Bug"
    - "Improvement"

workflow:
  source_of_truth: "jira-only"|"hybrid"

initialized_at: "YYYY-MM-DDTHH:MM:SSZ"
```

### Phase 6: Update AGENTS.md

1. Read `AGENTS.md` in the project root (if it exists)
2. **If it already references trazador:** Do not duplicate — skip
3. **If AGENTS.md exists but has no trazador reference:** Append a trazador section
4. **If AGENTS.md does not exist:** Ask to create one

### Phase 7: Validation & Summary

1. **If Jira MCP is available**, run a quick validation:
   - Fetch one issue from the project to confirm access
   - Report: "Validated: can read/write issues in <project>."

2. **Present the summary:**

```
Trazador initialized successfully.

Configuration:
  PM Tool:        Jira
  Project:        <project_name> (<project_key>)
  Source of truth: <jira-only|hybrid>
  Jira MCP:       connected|not configured
  Config file:    .trazador/config.yaml

Available commands:
  /trazador:discovery  — Analyze project state, find gaps, define scope
  /trazador:plan       — Define a feature and create Jira issues
  /trazador:work       — Pick up an issue and implement it
  /trazador:review     — Validate implementation against spec
  /trazador:sync       — Check project health

Ready to go. Run /trazador:discovery to assess your project, or /trazador:plan to start planning.
```

## Important Guidelines

- **NEVER write code** — this command only configures the workspace
- **Non-destructive** — never overwrite existing AGENTS.md content, only append
- **Idempotent** — running init twice updates config, doesn't create duplicates
- **Graceful degradation** — works even without Jira MCP (config-only mode)
- **One question at a time** — use AskUserQuestion sequentially, not in batches
