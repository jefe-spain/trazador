# Trazador — Jira-Native Agent Workflow

Trazador connects Claude/Codex agents to Jira so that every task is traceable from requirement to shipped code.

## Core Principle

**Jira is the single source of truth.** No local docs/plans/ or docs/brainstorms/ files.
All specs, plans, progress, and reviews live as Jira issues and comments.

## Prerequisites

Trazador bundles the **Jira MCP server** (`jira-server`) via `.mcp.json`.
On first use, Jira's OAuth flow will prompt you to authorize via browser.
If the MCP is not loading, `/trazador:init` will help you configure it.

## Configuration

Project-specific settings live in `.trazador/config.yaml`, generated by `/trazador:init`.

All commands read workspace constants (project, statuses, labels) from this config file.
**You must run `/trazador:init` once per project before using other commands.**

## Commands

| Command | Purpose | When to use |
|---------|---------|-------------|
| `/trazador:init` | One-time setup — configure Jira workspace, validate MCP | First time using trazador in a project |
| `/trazador:discovery` | Project reality check — analyze code vs Jira, find gaps | Periodic assessment |
| `/trazador:plan` | PM intake — define a feature, create Jira issues | New feature idea or goal |
| `/trazador:work` | Agent execution — pick up a Jira issue and implement it | A Todo issue is ready |
| `/trazador:review` | Goal-aligned review — validate work against spec | PR is ready |
| `/trazador:sync` | State sync — update Jira from agent actions | Automatic or manual |
| `/trazador:autopilot` | Autonomous supervisor — dispatch agents per issue | Hands-free execution |

## Agent Session Protocol

Every agent session MUST:

1. Read `.trazador/config.yaml` for workspace constants
2. Check Jira for assigned/in-progress issues
3. If resuming work, read the issue and its comments for context
4. Never create local execution trackers — use Jira
5. On completion, always sync to update Jira state

## Reading Config

All commands must read `.trazador/config.yaml` at the start to get:
- `jira.project_name` and `jira.project_key` — the Jira project
- `jira.statuses.*` — mapped status names
- `jira.labels` — available labels
- `workflow.source_of_truth` — "jira-only" or "hybrid"

**Never hardcode project names.** Always read from config.

## Design Principles

1. **Stateless agents** — Any session can pick up any issue. Context comes from Jira.
2. **PM stays in control** — Agents never auto-assign without explicit trigger.
3. **Simplest path** — Minimal implementation that meets acceptance criteria.
4. **Goal alignment** — Every review validates against the original issue spec.
5. **No local artifacts** — Plans, specs, and progress live in Jira. Code lives in git.
