# Trazador Init — Project Setup Wizard

One-time interactive setup that binds this repository to a Linear workspace and configures the trazador workflow.

## Input

<init_input> #$ARGUMENTS </init_input>

Arguments are ignored — this command is always interactive.

## Pre-flight Check

Before starting, check if `.trazador/config.yaml` already exists:

1. Read `.trazador/config.yaml`
2. If it exists and has valid content:
   - Show the current configuration to the user
   - Ask: "Trazador is already configured. Do you want to **reconfigure** or **exit**?"
   - If exit, stop here
   - If reconfigure, proceed with the wizard (existing values become defaults)

## Execution Flow

### Phase 1: Ensure Linear MCP is Available

Trazador requires the Linear MCP server (`linear-server`). This phase ensures it's configured and working.

**Step 1: Check if the MCP is loaded**

The trazador plugin bundles Linear MCP via its `.mcp.json`. It should be available automatically.

1. Attempt to call `mcp__linear-server__list_teams()`
2. **If it succeeds:**
   - Store the available teams
   - Tell the user: "Linear MCP is connected. Found N team(s)."
3. **If it fails:**
   - The bundled MCP may not have loaded, or OAuth hasn't been completed.
   - Tell the user:
     ```
     Linear MCP is not responding.

     The trazador plugin bundles the Linear MCP server, but it may need authorization.
     On first use, you'll be prompted to authorize via browser.

     If the plugin MCP didn't load, you can add it manually:
       claude mcp add --transport http linear-server https://mcp.linear.app/mcp

     Or add to your project's .mcp.json:
       { "mcpServers": { "linear-server": { "type": "http", "url": "https://mcp.linear.app/mcp" } } }
     ```
   - Ask: "Would you like to retry, continue without Linear (config only), or stop?"
   - If retry, go back to the test
   - If continue, mark `linear_mcp_verified: false` and skip to Phase 5
   - If stop, exit gracefully

### Phase 2: Select Linear Workspace

**If MCP is available and multiple teams exist:**

1. Present the list of teams to the user
2. Ask: "Which team will this project use?" (single choice)
3. Fetch the selected team's details: `mcp__linear-server__get_team(name: <selected>)`

**If only one team:**
- Auto-select it, confirm with user: "Using team: <name> (<key>). Correct?"

**Store:** `team_name`, `team_key`

### Phase 3: Select Linear Project

1. Fetch projects for the selected team: `mcp__linear-server__list_projects(team: <team_name>)`
2. **If projects exist:**
   - Present the list
   - Ask: "Which project should trazador track?" (single choice)
   - Include option: "Create a new project"
3. **If no projects or user wants new:**
   - Ask for the project name
   - Create it: `mcp__linear-server__save_project(name: <name>, teamIds: [<team_id>])`

**Store:** `project_name`, `project_id`

### Phase 4: Verify Statuses & Labels

1. Fetch team workflow states: `mcp__linear-server__list_issue_statuses(team: <team_name>)`
2. Verify these statuses exist: `Backlog`, `Todo`, `In Progress`, `In Review`, `Done`
   - If any are missing, warn the user and list what's available
   - Map the closest equivalents (e.g., "To Do" → "Todo")
3. Fetch labels: `mcp__linear-server__list_issue_labels(team: <team_name>)`
4. Check for: `Feature`, `Bug`, `Improvement`
   - If missing, ask: "Should I create the missing labels?"
   - If yes, create them via `mcp__linear-server__create_issue_label`

**Store:** `statuses` (mapped names), `labels` (available labels)

### Phase 5: Configure Source of Truth

Ask the user with **AskUserQuestion**:

> How should this project manage specs and plans?
>
> 1. **Linear-only** (recommended) — All specs, plans, and progress live in Linear. No local docs/plans/ files.
> 2. **Hybrid** — Linear for tracking, but allow local docs/ for architecture notes and research.

**Store:** `source_of_truth: "linear-only" | "hybrid"`

### Phase 6: Write Configuration

Create `.trazador/config.yaml` with:

```yaml
# Trazador Configuration
# Generated by /trazador:init on YYYY-MM-DD
# Docs: See trazador plugin SKILL.md

linear:
  mcp_verified: true|false
  team_name: "<team>"
  team_key: "<KEY>"
  project_name: "<project>"
  project_id: "<id>"
  statuses:
    backlog: "Backlog"
    todo: "Todo"
    in_progress: "In Progress"
    in_review: "In Review"
    done: "Done"
  labels:
    - "Feature"
    - "Bug"
    - "Improvement"

workflow:
  source_of_truth: "linear-only"|"hybrid"

initialized_at: "YYYY-MM-DDTHH:MM:SSZ"
```

### Phase 7: Update AGENTS.md

1. Read `AGENTS.md` in the project root (if it exists)
2. **If it already references trazador or `.trazador/config.yaml`:**
   - Do not duplicate — skip this step
3. **If AGENTS.md exists but has no trazador reference:**
   - Append a trazador section:

```markdown

## Trazador Workflow

This project uses [trazador](https://github.com/jefernandez/trazador) for Linear-native agent workflow.

- **Config:** `.trazador/config.yaml`
- **Commands:** `/trazador:init`, `/trazador:discovery`, `/trazador:plan`, `/trazador:work`, `/trazador:review`, `/trazador:sync`
- **Source of truth:** Linear
- **Session protocol:** Read `.trazador/config.yaml`, check Linear for active issues, update Linear on completion.
```

4. **If AGENTS.md does not exist:**
   - Ask the user: "No AGENTS.md found. Should I create one with trazador workflow guidelines?"
   - If yes, create a minimal AGENTS.md with the trazador section above

5. **Do NOT modify** any existing content in AGENTS.md — only append.

### Phase 8: Validation & Summary

1. **If Linear MCP is available**, run a quick validation:
   - Fetch one issue from the project to confirm access: `mcp__linear-server__list_issues(team: <team>, project: <project>)`
   - Report: "Validated: can read/write issues in <project>."

2. **Present the summary:**

```
Trazador initialized successfully.

Configuration:
  Team:           <team_name> (<team_key>)
  Project:        <project_name>
  Source of truth: <linear-only|hybrid>
  Linear MCP:     connected|not configured
  Config file:    .trazador/config.yaml

Available commands:
  /trazador:discovery  — Analyze project state, find gaps, define scope
  /trazador:plan       — Define a feature and create Linear issues
  /trazador:work       — Pick up an issue and implement it
  /trazador:review     — Validate implementation against spec
  /trazador:sync       — Check project health

Ready to go. Run /trazador:discovery to assess your project, or /trazador:plan to start planning.
```

## Important Guidelines

- **NEVER write code** — this command only configures the workspace
- **Non-destructive** — never overwrite existing AGENTS.md content, only append
- **Idempotent** — running init twice updates config, doesn't create duplicates
- **Graceful degradation** — works even without Linear MCP (config-only mode)
- **One question at a time** — use AskUserQuestion sequentially, not in batches
